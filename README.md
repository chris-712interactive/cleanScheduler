# cleanScheduler

Multi-tenant scheduling, quoting, billing, and customer-service platform for
residential and commercial cleaning businesses. Three portals served from a
single Next.js deployment, switched by subdomain:

| Subdomain                        | Portal              |
|----------------------------------|---------------------|
| `admin.<apex>`                   | Founder admin       |
| `my.<apex>`                      | Unified customer    |
| `<tenant-slug>.<apex>`           | Tenant operations   |
| `<apex>` / `www.<apex>`          | Marketing site      |

## Tech stack

- **Next.js 15** (App Router, RSC)
- **TypeScript**, strict
- **Supabase** (Postgres + Auth + Storage + Realtime + Edge Functions) with
  Row-Level Security as the multi-tenancy boundary
- **SCSS Modules** + design-token system (no Tailwind)
- **Radix UI** primitives for accessible, headless component behavior
- **Lucide** icons
- **Zod** for env / schema validation
- **Vercel** for hosting (wildcard domains; cron triggers Edge Functions)

See [`.cursor/docs/plan/implementation-plan.md`](.cursor/docs/plan/implementation-plan.md)
for the full plan, and [`docs/design/portal-mockups/`](docs/design/portal-mockups)
for portal reference screens.

## Running locally

### 1. Install dependencies

```bash
pnpm install      # or: npm install
```

### 2. Configure env

```bash
cp .env.example .env.local
```

Fill in the Supabase URL + anon key + service role key from your local
Supabase project. Set `NEXT_PUBLIC_APP_DOMAIN=lvh.me:3000` so wildcard
subdomains resolve to `127.0.0.1` automatically.

### 3. Run the dev server

```bash
pnpm dev          # or: npm run dev
```

Visit each portal:

- Marketing: <http://lvh.me:3000>
- Admin: <http://admin.lvh.me:3000>
- Customer: <http://my.lvh.me:3000>
- Tenant (any slug): <http://acme.lvh.me:3000>

### Useful scripts

```bash
pnpm typecheck         # tsc --noEmit
pnpm lint              # next lint
pnpm lint:styles       # stylelint **/*.scss
pnpm format            # prettier --write .
pnpm build             # next build
```

## Color system

- Brand palette is captured in `styles/tokens/_colors.scss` (the only place
  hex literals are allowed - stylelint enforces this).
- Theme layer (`styles/_theme.scss`) maps tokens to semantic CSS custom
  properties (`--color-bg`, `--color-text`, `--color-brand-primary`, ...).
- Pre-hydration script (`components/theme/themeScript.ts`) sets
  `data-theme="light|dark"` on `<html>` *before* React hydrates, so users
  never see a flash of the wrong theme.
- `<ThemeToggle />` exposes a System / Light / Dark control persisted to
  `localStorage.cs_theme`.

## Multi-tenancy

- Middleware classifies the incoming host into one portal kind and forwards
  `x-portal` (+ `x-tenant-slug` when applicable) headers to downstream Server
  Components.
- It also rewrites the URL internally to a portal-prefixed path
  (`/admin/...`, `/tenant/...`, etc.) so each portal owns a distinct route
  tree without collisions.
- RLS policies (added in a follow-up migration) will enforce tenant isolation
  at the database layer; the JWT carries `current_tenant_id` + `tenant_role`
  claims for that purpose.
